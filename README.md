## Adding or changing a monitor

All configuration lives in [`.upptimerc.yml`](./.upptimerc.yml). To monitor a new
service, add an entry to the `sites` list and commit to the default branch — the
workflows pick up the change on the next run. See the
[Upptime configuration docs](https://upptime.js.org/docs/configuration) for the
full set of options (custom headers, expected status codes, content assertions,
notifications, and more).

## How it works

- **Uptime CI** runs every 5 minutes and pings each endpoint. If a check fails, it
  opens a GitHub Issue as an incident and closes it automatically when the service
  recovers.
- **Response Time CI** records latency and commits it to git, building a historical
  trend over time.
- **Graphs CI** renders the response-time charts shown on the status page.
- **Static Site CI** builds the Svelte status page and deploys it to GitHub Pages.
- **Summary CI** keeps this README's status table in sync.

## Setup checklist (one-time)

These steps are done in the GitHub repository settings, not in code:

1. **GitHub Pages** — Settings → Pages → deploy from the `gh-pages` branch (created
   automatically by the Static Site CI on its first run).
2. **Custom domain** — add a DNS `CNAME` record `status` → `delopay-net.github.io`,
   so the page resolves at `status.delopay.net` (already set as `cname` in the config).
3. **`GH_PAT` secret** _(recommended)_ — a fine-grained personal access token with
   `contents`, `issues`, and `pages` write access, stored as the `GH_PAT` Actions
   secret. The workflows fall back to the default `GITHUB_TOKEN` if it is absent,
   but a dedicated PAT avoids rate limits and lets incident issues be labelled and
   assigned correctly.
